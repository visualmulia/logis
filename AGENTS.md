# Logis — Agent Reference

## Project Overview

**Logis** is a Progressive Web App (PWA) for construction project administration and logistics in Indonesia. It is built with Next.js 16 (App Router), React 19, TypeScript, and Firebase. The entire user interface is in **Bahasa Indonesia**; comments and identifiers in source code are also predominantly Indonesian.

The application supports multi-tenant companies, role-based access control, real-time data via Firestore, push notifications, and offline-capable PWA installation.

### Primary Modules

1. **Request Material** — Field teams request materials; PMs acknowledge; central admins approve; POs are issued; logistics confirms delivery.
2. **Gudang Digital (Inventory)** — Per-project stock tracking for materials, APD, consumables, and tools with low-stock alerts.
3. **Equipment Tracker** — Asset registration, transfers, condition tracking, and service schedule monitoring.
4. **Petty Cash** — Field cash requests with approval workflow, anomaly detection, receipt upload, and online purchase tracking.
5. **Reports & Overview** — Dashboard stats, recent activity, and PDF export for requests, petty cash, and inventory.

## Code Style Guidelines

- **Language**: UI text and most inline comments are in **Bahasa Indonesia**. When adding new UI labels, use Indonesian.
- **Styling**: Tailwind CSS utility classes are used extensively. For one-off colors that don't fit the theme, inline `style` props are common (especially orange `#F97316`).
- **Theming**: The app supports light and dark modes via CSS custom properties (`--bg-primary`, `--text-primary`, etc.) managed in `ThemeContext.tsx`. shadcn/ui tokens are defined in `globals.css`.
- **Path alias**: Use `@/` for imports from `src/` (configured in `tsconfig.json`).
- **Components**: shadcn/ui components live in `src/components/ui/`. New components should follow the existing patterns (function components, explicit return types optional but types are strict).
- **Firebase**: All Firebase SDK access goes through `src/lib/firebase/config.ts`. Do not initialize Firebase in multiple places.

## Role-Based Access Control

The app has 7 roles. The `logisUser.role` field determines what a user can see and do.

| Role | Description | Typical Access |
|------|-------------|----------------|
| `owner` | Company owner | Full access |
| `admin` | Central admin | Request material approval, PO issuance |
| `pm` | Project Manager | Acknowledge/review requests, view assigned projects only |
| `supervisor` | Site supervisor | Create requests, view inventory |
| `logistik` | Logistics | Inventory, assets, delivery confirmation |
| `admin_site` | Site admin | Petty cash, inventory |
| `readonly` | Read-only | View only |

The `Sidebar.tsx` filters navigation items by role. The `OverviewPage` also adapts stats and quick actions per role.

## Security Considerations

1. **Firebase Rules**: Firestore and Storage security rules must be configured in the Firebase Console. The app assumes rules enforce company-level isolation.
2. **API Route**: `api/send-notification` implements IP-based rate limiting (10 requests/minute) and Zod validation before sending FCM messages.
3. **Service Account**: `FCM_SERVICE_ACCOUNT` is server-side only and must never be exposed to the client.
4. **Middleware**: `middleware.ts` currently passes through all requests but is wired to run on all non-static routes.
5. **Route Guard**: `RouteGuard` component redirects unauthenticated users to `/login` on the client side.

## PWA & Offline

- The app is a PWA configured via `@ducanh2912/next-pwa` with output to `public/`.
- `manifest.json` defines the app as `standalone` with an orange (`#F97316`) theme.
- An `offline/page.tsx` exists for offline fallback.
- PWA registration is handled by `PWARegister` component in the root layout.

## PDF Export

`src/lib/pdf/exportPDF.ts` provides three generators:

All PDFs use a dark header with an orange accent bar and Indonesian date formatting.

## Common Pitfalls for Agents

- **Next.js 16 specifics**: Check `node_modules/next/dist/docs/` if an API behaves differently from your training data.
- **Firebase long polling**: Firestore is initialized with `experimentalForceLongPolling: true`.
- **Company resolution**: User data is stored under `logis_companies/{companyId}/users/{uid}`. The `AuthContext` resolves `companyId` via `user_company_index` and localStorage.
- **PM scope**: Project Managers (`pm`) are scoped to `projectIds` assigned to them. Always filter Firestore queries by `projectIds` for PM users.
- **Indonesian copy**: All user-facing strings must remain in Bahasa Indonesia.
